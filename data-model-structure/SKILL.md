## 术语定义

| 术语 | 定义 | 示例 |
|------|------|------|
| **实体** | 数据库表对应的业务对象 | Order、User |
| **字段** | 表中的列 | id、name、amount |
| **索引** | 提高查询效率的数据结构 | PRIMARY KEY、INDEX |
| **枚举** | 固定值的集合 | StatusEnum(0,1,2) |
| **关联关系** | 表与表之间的引用关系 | 1:N、N:M |

## 输入文档

| 文档类型 | 用途 | 必须 |
|----------|------|------|
| PRD 功能需求 | 实体的业务含义和关系 | 是 |
| 架构文档 | 数据架构设计 | 否（参考用） |

## 输出结构

```
data-model/
├── INDEX.md                      # 总文档：索引 + 公共规范 + 表关联关系
├── 10-概述/
│   └── 10-模型概述.md           # 命名规范、字段定义规范
├── 20-XXX模块/
│   ├── 20-t_user.md             # 用户表
│   └── 20-t_order.md            # 订单表
├── 30-XXX模块/
│   ├── 30-t_product.md          # 产品表
│   └── 30-t_category.md         # 分类表
└── ...
```

> ⚠️ **重要原则：一个文件只包含一张表**，每个表独立成文，文件名与表名一致。

## 核心原则

### 1. 总-分结构
- **总文档（INDEX.md）**：索引导航 + 公共规范 + 表关联关系
- **分文档（按模块划分）**：各模块数据表独立成文，职责单一

### 2. 领域划分
按业务模块划分文档，而非按技术类型划分：

| 文档类型 | 职责 | 不包含 |
|----------|------|--------|
| 模型概述.md | 通用规范（命名、字段、约束） | 具体表结构 |
| XX-t_xxx.md | **一张表一份文档**，独立完整 | 通用规范 |

### 3. 单一职责

| 文档 | 职责 |
|------|------|
| INDEX.md | 索引导航 + 公共规范 + 表关联关系 |
| 10-模型概述.md | 通用规范（命名、字段、约束） |
| XX-t_xxx.md | 一张表一份文档，独立完整 |

> ✅ **每个表文件必须职责单一**：只描述一张表的所有细节（字段、索引、枚举、关联），不混入其他表的内容。

## 二、输入处理与文档理解

1. **获取原始文档**
   用户通常直接粘贴 PRD/需求文档文本。若用户提供的是截图，应礼貌建议将图中文字转为纯文本后再继续。

2. **识别实体与关系**
   从文档中提取：业务实体、实体属性、表关联关系（1:N、N:M）。

3. **记录关键约束**
   - 必填/可选字段
   - 格式、长度、范围限制
   - 业务规则
   - 状态迁移规则
   - 数据依赖关系

## 三、文档创建 SOP

### 3.1 创建总文档（INDEX.md）

总文档是数据模型的门户文档，提供全局数据视图和导航。

必须包含：
- 文档信息（版本、日期、修订历史）
- 索引导航（各模块文档链接）
- 通用规范（命名规范、字段定义）
- 表关联关系（ER图或表格）
- 关联表汇总

#### 表关联关系

**ER图格式（推荐）**：

```mermaid
erDiagram
    实体A ||--o{ 实体B : "1:N"
    实体B ||--o{ 实体C : "1:N"
```

**表格格式**：

| 主表 | 从表 | 关系 | 外键字段 | 说明 |
|------|------|------|----------|------|
| 实体A | 实体B | 1:N | entity_a_id | 一个A多个B |

#### 关联表汇总

| 表名 | 说明 | 关联表 |
|------|------|--------|
| t_user | 用户表 | t_order |
| t_order | 订单表 | t_order_detail |

### 3.2 创建模型概述（10-概述/10-模型概述.md）

必须包含：
- **命名规范**：表命名、字段命名规则
- **字段定义规范**：类型选择、约束规则
- **通用字段**：id、dr、create_time、update_time
- **主键策略**：自增/雪花算法/UUID
- **索引设计**：索引类型、选择原则
- **约束规范**：NOT NULL、UNIQUE、DEFAULT、COMMENT

#### 表命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 业务表 | t_模块名 | t_user, t_order |
| 关联表 | t_表A_表B_rel | t_user_role_rel |
| 历史表 | t_表名_history | t_order_history |
| 日志表 | t_log_业务名 | t_log_operation |

#### 字段命名规范

| 类型 | 规范 | 示例 |
|------|------|------|
| 主键 | id | id |
| 外键 | 表名_id | entity_a_id |
| 删除标记 | dr | dr |
| 创建时间 | create_time | create_time |
| 更新时间 | update_time | update_time |
| 状态 | status | status |
| 金额 | xxx_amount | order_amount |

#### 字段定义规范

| 数据类型 | 使用场景 | 示例 |
|----------|----------|------|
| ID | 主键、外键 | BIGINT |
| 名称 | 文字描述 | VARCHAR(100) |
| 代码 | 编码、编号 | VARCHAR(50) |
| 金额 | 钱相关 | DECIMAL(12,2) |
| 布尔 | 是/否开关 | TINYINT |
| 日期 | 日期 | DATETIME |
| 枚举 | 状态、类型 | TINYINT |
| JSON | 结构化数据 | JSON |

#### 通用字段

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT | 主键 |
| dr | TINYINT | 删除标记（0-未删除 1-已删除） |
| create_time | DATETIME | 创建时间 |
| update_time | DATETIME | 更新时间 |

#### 主键策略

| 策略 | 说明 | 适用场景 |
|------|------|----------|
| 自增主键 | 数据库自动增长 | 单表插入 |
| 雪花算法 | 分布式ID生成 | 分库分表 |
| UUID | 字符串唯一ID | 分布式系统 |

#### 索引设计

| 类型 | 命名 | 说明 |
|------|------|------|
| PRIMARY KEY | pk_ | 主键索引 |
| UNIQUE | uk_字段名 | 唯一索引 |
| INDEX | idx_字段名 | 普通索引 |

### 3.3 按模块按表创建数据表文档

每个表独立成文，文件名格式：`{序号}-{表名}.md`

例如：`20-t_user.md`、`30-t_order_detail.md`

每个表文档包含：

| 内容 | 说明 |
|------|------|
| 表基本信息 | 表名、中文名、说明 |
| 表结构 | 字段、类型、约束、说明 |
| 枚举定义 | 值、名称、说明 |
| 索引 | 索引名、字段、类型 |
| 关联关系 | 表名、关系、说明 |

> ⚠️ **禁止多表混装**：一个 .md 文件只描述一张表，不允许多张表写在同一个文件里。

## 四、数据表文档详细规则

### 4.1 表基本信息

```markdown
# t_user 用户表

## 表基本信息

| 属性 | 值 |
|------|------|
| 表名 | t_user |
| 中文名 | 用户表 |
| 说明 | 存储系统用户信息 |
```

### 4.2 表结构

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | BIGINT | PK | 主键ID |
| name | VARCHAR(100) | NOT NULL | 用户姓名 |
| code | VARCHAR(50) | NOT NULL, UNIQUE | 用户编码 |
| status | TINYINT | DEFAULT 1 | 状态 |
| dr | TINYINT | DEFAULT 0 | 删除标记 |
| create_time | DATETIME | NOT NULL | 创建时间 |
| update_time | DATETIME | NOT NULL | 更新时间 |

### 4.3 枚举定义

| 值 | 名称 | 说明 |
|----|------|------|
| 1 | NORMAL | 正常 |
| 2 | DISABLED | 禁用 |

### 4.4 索引

| 索引名 | 字段 | 类型 | 说明 |
|--------|------|------|------|
| PRIMARY | id | PRIMARY | 主键索引 |
| uk_code | code | UNIQUE | 唯一索引 |

### 4.5 关联关系

| 表名 | 关系 | 说明 |
|------|------|------|
| t_order | 1:N | 一个用户多个订单 |

## 五、代码注释规范

每个字段必须包含说明，用于生成代码注释：

```markdown
| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|
| id | BIGINT | PK | 主键ID，用户唯一标识 |
| name | VARCHAR(100) | NOT NULL | 用户姓名，最大100字符 |
```

## 六、技术栈无关原则

- 不指定具体数据库类型（MySQL/PostgreSQL）
- 不指定具体 Java 类型（Long/BigDecimal）
- 字段类型使用通用数据库类型描述

## 八、生成文档的保存与落盘

- **默认行为**：生成的数据模型文档（以"总-分结构"展示：`INDEX.md` + 各模块文件）。**必须**保存到目标路径下的 `data-model/` 目录，并在其内部按文档中描述的"总-分结构"创建与写入文件。
  - 默认保存到**当前工作区根目录**的 `data-model/`。
  - 若用户给的是目录：落盘到 `<用户目录>/data-model/`。
  - 若用户给的是文件夹但不含 `data-model/`：自动补齐为 `<用户文件夹>/data-model/`。
  - `data-model/` 内的类型根目录： `data-model/`；**内部结构仍需遵循"总-分结构"**（包含 `INDEX.md` 与按模块拆分的子目录/文件）。

  示例：
	```
	data-model/
	├── INDEX.md                      # 总文档：索引 + 公共规范 + 表关联关系
	├── 10-概述/
	│   └── 10-模型概述.md           # 命名规范、字段定义规范
	├── 20-用户模块/
	│   ├── 20-t_user.md             # 用户表
	│   └── 20-t_user_role.md        # 用户角色表
	├── 30-订单模块/
	│   ├── 30-t_order.md            # 订单表
	│   └── 30-t_order_detail.md     # 订单明细表
	└── ...
	```

- **自动执行写盘**：除非用户明确要求不保存，否则直接写入工作空间对应的目录。
